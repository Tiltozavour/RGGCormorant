# Cormorant Society: правила игровых механик

Этот документ описывает текущие игровые правила проекта так, как они реализованы в коде.

## 1. Игрок И Участник Партии

Зарегистрированный пользователь еще не считается участником текущей партии.

Пользователь становится участником партии только после выбора стартовой клетки:

- клетка `6`;
- клетка `15`.

Технически это означает:

```ts
inGame === true && role !== "admin"
```

При регистрации создается профиль игрока с `inGame: false`. После выбора стартовой клетки приложение выставляет:

- `position` - выбранная стартовая клетка;
- `prevCell: null`;
- `inGame: true`;
- стартовые карты в `inventory`.

Игроки без `inGame: true` не должны участвовать в очереди ходов, результатах, голосовании, таблицах и игровых таргетах.

## 2. Фазы Игры

Текущий порядок фаз:

```text
waiting_game -> playing -> results -> voting -> turn -> next_game
```

Назначение:

- `waiting_game` - ожидание или подготовка мини-игры.
- `playing` - мини-игра идет.
- `results` - админ вводит результаты мини-игры.
- `voting` - игроки голосуют за бонус.
- `turn` - игроки ходят по полю и используют карты.
- `next_game` - переход к следующему раунду и колесо выбора игры.

## 3. Результаты Раунда

В фазе `results` админ вводит очки мини-игры для участников партии.

После сохранения:

- игрок получает эти очки в общий баланс `tiltCoins`;
- результат мини-игры записывается в `lastTiltoCoins`;
- временный бонус голосования `bonusPoints` сбрасывается;
- `gameState.currentResults[playerId]` получает результат мини-игры.

Игроки с одинаковыми очками считаются равными. Отдельное место в данных не хранится.

## 4. Голосование

Голосование проходит в фазе `voting`.

Сейчас голосовать может игрок, который получил больше `0` очков за мини-игру:

```ts
lastTiltoCoins > 0
```

Голосовать можно за участника партии, который тоже получил больше `0` очков за мини-игру.

Победитель голосования определяется по количеству голосов, а не по очкам мини-игры.

Бонус:

- один победитель по голосам: `+3`;
- два победителя по голосам: `+2` каждому;
- три и больше победителей по голосам: `+1` каждому.

Бонус записывается в:

- `tiltCoins`;
- `bonusPoints`;
- `gameHistory.scores[playerId].voting`.

После завершения голосования `currentResults` обновляется до итогового результата раунда:

```text
очки мини-игры + бонус голосования
```

Именно эта сумма используется дальше для очереди хода и золотой карты.

## 5. Очередь Ходов

Очередь строится при переходе в фазу `turn`.

В очередь попадают только участники партии с итогом раунда больше `0`.

Порядок:

1. Сначала игроки сортируются по итоговым очкам раунда по убыванию.
2. Если очки равны и оба игрока уже были в прошлой очереди, сохраняется их прошлый относительный порядок.
3. Если прошлой очереди для игроков нет, порядок между ними определяется случайно.

Пример:

```text
Аня: 10
Боря: 10
Катя: 5
```

Аня и Боря будут первыми в случайном порядке, Катя пойдет после них.

Игроки с `0` или отрицательным итогом не попадают в очередь автоматически. Админ может добавить таких игроков вручную из панели хода.

## 6. Золотая Карта

Золотая карта - это временный пассивный бонус, который дает скидку `50%` в B-Shop.

Она не хранится как обычная карта в инвентаре. В интерфейсе она отображается виртуально через `inv_018`, если id игрока есть в:

```ts
gameState.goldenCardHolderIds
```

Кто получает золотую карту при переходе в фазу `turn`:

- все участники с итогом раунда `<= 0`;
- плюс один последний игрок среди участников с итогом больше `0`.

Последний игрок определяется после построения очереди ходов. Если есть ничья, сначала применяется правило очереди, включая случайный tie-breaker.

Скидка считается так:

```ts
Math.ceil(basePrice / 2)
```

Минимальная цена после скидки - `1`.

Примеры:

- `3 -> 2`;
- `7 -> 4`;
- `15 -> 8`.

Скидка применяется только при покупке карты в B-Shop.

## 7. Поле И Движение

Поле состоит из клеток с типом:

- `neutral`;
- `b-shop`;
- `gambling`.

Игрок движется по броску кубика. На развилках игрок выбирает направление.

После завершения движения:

- обычная клетка завершает ход;
- `b-shop` открывает B-Shop interaction;
- `gambling` открывает Gambling interaction.

Некоторые карты могут двигать самого игрока или чужую фишку. Для чужого движения используется отдельное состояние `cardMove`.

## 8. B-Shop

B-Shop открывается при попадании на клетку `b-shop`.

Игрок видит набор карт на покупку. Цена берется из карты или из дефолта по редкости:

- `common`: `3`;
- `rare`: `7`;
- `epic`: `15`;
- `legendary`: `0`, но легендарки обычно идут через призовую механику.

Если у игрока активна золотая карта, цена уменьшается на `50%` с округлением вверх.

Игрок может:

- купить карту, если хватает монет;
- уйти без покупки.

## 9. Gambling

Gambling открывается при попадании на клетку `gambling`.

Игрок выбирает одну из закрытых карт. После выбора карта раскрывается и применяется.

В Gambling могут попадаться:

- обычные карты;
- моментальные эффекты;
- с небольшим шансом легендарная призовая карта, если она еще не была выиграна.

Если у игрока есть подходящая защита, он может использовать ее вместо негативного эффекта.

## 10. Карты

Основные группы карт:

- добавление или потеря монет;
- движение вперед или назад;
- перемещение другого игрока;
- телепорт;
- защита;
- отражение эффекта;
- кража монет;
- кража или сброс карты;
- налоги;
- дуэль;
- повтор колеса;
- легендарные призы.

Обычные карты лежат в Firestore collection `cards`.

Легендарные призовые карты лежат в `prizes` и считаются уникальными:

- `isUnique: true`;
- `isWon: true`;
- `winnerId` хранит победителя.

Повторная выдача уже выигранной легендарной карты блокируется.

## 11. Защиты И Отражение

В проекте есть несколько защитных механик:

- обычная защита от Gambling;
- Mr. Fish как универсальная защита;
- отражение карты обратно в атакующего.

Отражение работает не на все карты. Список отражаемых карт задан в коде.

Если эффект отражается, цель и инициатор могут поменяться местами с точки зрения применения эффекта.

## 12. Налоги

Карта налогов запускает цепочку ответов для выбранных игроков.

Игроки могут:

- заплатить монеты;
- применить подходящую защитную карту;
- выбрать Gambling-ответ, если он доступен логикой карты.

Налоговый банк собирается и затем уходит получателю через `pendingTaxPayout`.

## 13. Дуэли

Дуэль проходит через состояния:

```text
pending -> accepted -> betting -> ready_to_roll -> rolling -> admin_wait -> finished
```

В дуэли участвуют инициатор и цель.

Дуэль может быть:

- через кубики;
- через игру по выбору/решение админа.

Ставки и итоговые выплаты выполняются через транзакции Firestore.

## 14. Колесо Игры

Колесо выбирает следующую мини-игру из collection `wheel`.

После подтверждения результата:

- выбранная игра становится `active: false`;
- `currentGame` обновляется;
- состояние колеса сбрасывается;
- игра возвращается в фазу `waiting_game`.

Некоторые карты могут перезапускать или отменять результат колеса.

## 15. Админские Вмешательства

Админ может:

- переключать фазы;
- вводить результаты;
- завершать голосование;
- запускать колесо;
- сбрасывать состояние;
- выдавать карты;
- изменять монеты;
- вручную добавлять игроков с `0` очков в очередь хода;
- решать кастомные дуэли.

Админские действия нужны для гибкости живой партии, но они могут менять баланс. Для важных вмешательств желательно смотреть event log.

## 16. Важные Нюансы Баланса

- Игрок без стартовой клетки не участвует в партии.
- Игрок с `0` очков не попадает в очередь автоматически, но получает золотую карту.
- Последний положительный игрок тоже получает золотую карту.
- При равных очках порядок может быть случайным, если прошлой очереди нет.
- Голосование влияет на итог раунда и может изменить очередь ходов.
- Золотая карта влияет только на B-Shop, а не на Gambling или другие покупки/эффекты.
