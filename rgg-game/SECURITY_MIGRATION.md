# Firestore Hardening And Backend Migration

## Current Rules Hardening

- `gameEvents` create is schema-checked in `firestore.rules`: only known fields, known event types, bounded string sizes, optional `details` as a map.
- `invites` update is limited to claiming an unused invite with `used: true` and `usedBy: request.auth.uid`; admin can still manage invites.
- Admin-only collections remain admin-only: `cards`, normal `prizes` management, `wheel`, reset-style writes.

## Still Client-Side

These areas still need broad enough rules because the client currently performs the business operation directly:

- `players` runtime fields: coins, inventory, position, statuses, notifications.
- `gameState` runtime fields: turn state, wheel state, active interactions, duels, revealed cards, hot coin gains.
- Card effects that touch multiple documents.
- Special cells such as gambling and b-shop.

## Migration Priority

1. Move card play resolution to callable backend operations: `playCard`, `respondToCard`, `resolveInteraction`.
2. Move coin and inventory mutations behind backend validation.
3. Move movement resolution behind backend validation: dice roll, selected path, special cell opening.
4. Move duel resolution behind backend validation.
5. After each migration, remove the matching client-write allowance from `firestore.rules`.

## First Callable: `playCard`

Shared contract lives in `src/shared/playCardContract.ts`.

Callable name:

- `playCard`

Request:

- `version`: contract version, currently `1`.
- `cardId`: card being played.
- `targetPlayerId`: optional selected target.
- `source`: one of `hand`, `wheel`, `interaction`, `admin`.
- `clientRequestId`: idempotency key generated by the client.

Response:

- success: `{ ok: true, message?, openedInteractionId?, revealedCardId? }`
- failure: `{ ok: false, code, message }`

Backend responsibilities:

- Verify `context.auth.uid`.
- Load `players/{uid}`, `gameState/current`, card document from `cards` or `prizes`.
- Verify ownership unless source is an admin-only operation.
- Verify phase and turn restrictions.
- Verify target requirements and target validity.
- Apply all Firestore writes in one transaction.
- Remove exactly one matching card from inventory.
- Add the card to `revealedCards` where appropriate.
- Create/open any required `activeInteraction`.
- Write compact notifications or event logs if needed.

Recommended first migration slice:

1. Passive/simple self cards: protection, promo code, discard-next-drawn, take-next-card/give-next-card flags.
2. Wheel reroll cards: `inv_017` and fish cancellation around wheel.
3. Targeted interaction openers that do not immediately mutate target coins: discard selection, duel challenge start.
4. Coin and movement cards after their pure rule helpers are extracted.

Rules to tighten after `playCard` owns a slice:

- Remove matching direct self-updates from `players` for inventory/status fields.
- Remove matching `gameState` client-write fields such as `revealedCards`, `activeInteraction`, `rollBonus`, `cardDiceRoll` when no longer written directly by clients.

## Safe Intermediate Rule Policy

Until backend operations exist, avoid tightening `players` and `gameState` globally. Prefer small schema/flow restrictions that match existing client writes and can be manually tested right away.

## Current Client-Side Stabilization Status

As of 2026-05-15, the project is still client-driven, but many card flows have been moved from sequential `updateDoc` calls to transactions or transaction helpers.

New client helpers:

- `cardUseGuards.ts`
- `cardTargetRules.ts`
- `cardPlayPatches.ts`
- `cardPlayTransactions.ts`

Stabilized slices:

- Card spend plus `revealedCards`.
- Simple self effects: protection, promo code, discard-next-drawn.
- **Wheel stability:** Reroll and Fish cancellation moved to `runTransaction` boundary.
- Targeted openers: discard selection, steal-card selection, move-for-coins selection, duel challenge start, tax queue start.
- Coin and movement cards: add coins, steal coins, judge coins, communism, move steps, teleports, move-target-and-self.
- Parts of `handleReflectResponse`: reflect card spend, discard/move-for-coins transitions, judge, mage visual roll, Katjit, tax one-off response.

Remaining client-side audit candidates before backend migration:

1. `handleReflectResponse` branch `inv_007` opening `cardMove`.
2. `handleTaxResponse`.
3. `handleConfirmMoveForCoins`.
4. Gambling/B-Shop selection confirmation.
5. Duel response/status transitions.

This does not replace backend migration. It only reduces race conditions and makes the eventual callable migration easier because the desired transaction boundaries are now clearer.
